# Phase 0 カバレッジ検証（第一段：仕様ベース）
## Bedrock Guardrails 組み込みPII型 × RDDエンティティ 対応表

**目的:** ライブ実測の前に、Guardrails組み込みPII型の公式仕様（API Reference `GuardrailPiiEntityConfig`）とRDDの対象エンティティ（§3.1/§3.2）を突き合わせ、(1)規則層が守るべき範囲を確定し、(2)どのエンティティがどの層でも捕捉されない構造的ギャップかを特定する。

**確定事実:** Guardrails組み込みPII型は全31種。内訳は General / Finance / IT / USA固有 / Canada固有 / UK固有。**日本固有の識別子型は存在しない**。`NAME`型は組織名・住所の一部である名称を除外する旨が明記されている。

---

## 1. 組み込みPII型 全31種（公式）

General: ADDRESS, AGE, NAME, EMAIL, PHONE, USERNAME, PASSWORD, DRIVER_ID, LICENSE_PLATE, VEHICLE_IDENTIFICATION_NUMBER
Finance: CREDIT_DEBIT_CARD_CVV, CREDIT_DEBIT_CARD_EXPIRY, CREDIT_DEBIT_CARD_NUMBER, PIN, INTERNATIONAL_BANK_ACCOUNT_NUMBER, SWIFT_CODE
IT: IP_ADDRESS, MAC_ADDRESS, URL, AWS_ACCESS_KEY, AWS_SECRET_KEY
USA: US_BANK_ACCOUNT_NUMBER, US_BANK_ROUTING_NUMBER, US_INDIVIDUAL_TAX_IDENTIFICATION_NUMBER, US_PASSPORT_NUMBER, US_SOCIAL_SECURITY_NUMBER
Canada: CA_HEALTH_NUMBER, CA_SOCIAL_INSURANCE_NUMBER
UK: UK_NATIONAL_HEALTH_SERVICE_NUMBER, UK_NATIONAL_INSURANCE_NUMBER, UK_UNIQUE_TAXPAYER_REFERENCE_NUMBER
Custom: 正規表現（regex）

> 注：`DATE_OF_BIRTH`（生年月日）型は存在しない（`AGE`はあるが別物）。日本の郵便番号・銀行口座に対応する専用型も存在しない。

## 2. 対応表：RDDエンティティ × 捕捉層

凡例：✅=組み込みで捕捉可（文脈ML、日本語最適化対応）／△=部分・要実測／❌=組み込み型なし（規則層 or 別手段が必須）

### 2.1 §3.1（規則層想定：定型・日本固有識別子）

| RDDエンティティ | Guardrails組み込み | 規則層の要否 | 判定 |
|---|---|---|---|
| メールアドレス | ✅ EMAIL | 冗長化のため保持 | 規則∪Guardrailsの二重捕捉 |
| 電話番号 | △ PHONE（汎用、和式書式は要実測） | **必須** | 規則層で日本形式を確実捕捉 |
| マイナンバー（12桁） | ❌ 該当型なし | **必須（唯一の手段）** | 規則層のみ |
| 法人番号（13桁） | ❌ 該当型なし | **必須（唯一の手段）** | 規則層のみ |
| 銀行口座番号（日本形式） | ❌ US型のみ | **必須** | 規則層のみ |
| クレジットカード番号 | ✅ CREDIT_DEBIT_CARD_NUMBER | 冗長化のため保持 | 二重捕捉 |
| 郵便番号（〒・日本形式） | △ ADDRESS文脈で部分的 | **必須** | 規則層で確実捕捉 |
| 生年月日（和暦・西暦） | ❌ DOB型なし（AGEは別物） | **必須（唯一の手段）** | 規則層のみ |
| IPアドレス | ✅ IP_ADDRESS | 冗長化のため保持 | 二重捕捉 |
| URL | ✅ URL | 冗長化のため保持 | 二重捕捉 |

### 2.2 §3.2（マネージド検出層想定：文脈依存）

| RDDエンティティ | Guardrails組み込み | 判定 |
|---|---|---|
| 氏名（人名） | ✅ NAME（日本語最適化・文脈ML） | Guardrailsで捕捉。要実測（recall水準） |
| 住所 | ✅ ADDRESS | Guardrailsで捕捉。要実測 |
| **組織名・会社名** | ❌ **該当型なし**（NAMEは組織名を明示的に除外） | **構造的ギャップ。現設計では漏れる** |
| **役職** | ❌ **該当型なし** | **構造的ギャップ** |
| **施設名** | ❌ **該当型なし** | **構造的ギャップ** |

## 3. 重要な発見と設計含意

### 3.1 規則層は「補強」ではなく「日本固有構造化PIIの唯一の捕捉手段」
マイナンバー・法人番号・日本形式銀行口座・郵便番号・生年月日は、Guardrails組み込み型に一切存在しない。規則層がこれらの唯一の捕捉経路であり、RDD §3.1の必須性が確定的に裏付けられた。規則層の品質（パターン網羅・FP許容での広い捕捉）が、これらエンティティのrecallを直接決定する。

### 3.2 構造的ギャップ：組織名・会社名・役職・施設名（要意思決定）
Guardrailsの組み込みPII型には ORGANIZATION / JOB_TITLE / FACILITY に相当する型がない。RDD §3.2はこれらをGuardrailsに委ねる設計だったが、**現設計のままでは構造的に漏れる**。法務文書では組織名・会社名の機密性が高く、recall最優先方針と矛盾する。対応案は次の三つ。

- **案α：Comprehend汎用エンティティ認識を追加recall源にする**。Comprehendの一般NERは日本語対応で ORGANIZATION / LOCATION 等を返す（PII特化ではないが組織名・施設名の補完に有効）。和集合の第3層として追加。マネージドのまま完結し、自前モデル不要。**第一推奨**。
- **案β：Guardrailsの「禁止トピック（denied topics）」やカスタムで擬似的に捕捉**。エンティティ抽出には不向きで、recall・精度ともに不安定。非推奨。
- **案γ：機械検出では捕捉せず、全件人手レビューに委ねる**。組織名・役職・施設名はレビュー時に手動マスク。簡潔だがレビュー負荷が増大し、機械recall基準（§7）の前提が崩れる。

> いずれを採るかで、和集合の検出層数（2層か3層か）と、§6.3「補強層」の初期スコープ入り・スコープ外が変わる。RDD §3.2・§4.2・§6.3・§7 に波及する。

### 3.3 冗長捕捉はrecallにプラス
メール・電話・カード・IP・URLは規則層とGuardrailsの双方で捕捉されうる。和集合方針のもとでは二重捕捉は単純にrecall向上に寄与し、競合は§9の「広い方優先」で解決する。

## 4. 第二段：ライブ実測の設計（AWSアカウントで実行）

仕様ベースで判明しない点（実際の検出品質）は、利用者のAWSアカウントでの実測が必要。本環境からはAWSへ到達できないため、以下は実行設計（ハーネス仕様）として定義する。

### 4.1 実測で確かめるべき問い
1. `NAME` は日本語人名（漢字・ひらがな・カタカナ、姓名の分かち書き有無）をどの程度のrecallで検出するか。
2. `ADDRESS` は日本式住所（都道府県＋市区町村＋丁目番地、ビル名）をどの程度捕捉するか。
3. `PHONE` は日本の電話番号書式（市外局番、ハイフン有無、+81）を捕捉するか。
4. 組織名・役職・施設名が実際にどれだけ漏れるか（§3.2ギャップの実害測定）。
5. ANONYMIZE 時のタグ形式（`{NAME}` 等）と、スパン位置情報（trace）の取得可否。

### 4.2 検証用データ設計（実PII不使用）
- 合成日本語文書（契約書・通知書・社内文書を模した体裁）に、既知の正解スパン（人名・住所・各種番号・組織名・役職・施設名）を埋め込む。
- 各エンティティ型につき表記ゆれを網羅（漢字/かな、和暦/西暦、ハイフン有無 等）。
- 正解スパンのオフセットを保持し、検出結果と突合してエンティティ別recall・漏洩率を自動算出。

### 4.3 ハーネスの構成（実装は指示後）
- `ApplyGuardrail`（FM非呼出）で合成文書を評価 → 検出スパン取得。
- 規則層（正規表現）を独立に適用 → スパン取得。
- 和集合を取り、正解スパンと突合 → エンティティ別 recall / 漏洩率 / 過剰マスク率を出力。
- 出力をPhase 1のゲート基準（§7.4）と対照し、補強層（案α等）の要否を判定。

---

## 5. Phase 0 から確定する次アクション
1. §3.2ギャップへの対応方針（案α/β/γ）を決定 → RDDへ反映。
2. 規則層の確定パターン一覧を §3.1 ＋本表に基づき作成。
3. 上記4.2/4.3の実測ハーネスを（指示後に）実装し、利用者AWSで実行 → recall実測。
4. 実測結果でゲート基準（§7.4の0.98等）の現実妥当性を確認し、必要なら基準・補強を調整。
