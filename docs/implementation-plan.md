# 実装計画書
## 日本語文書 機密情報マスキングシステム（RDD完成版 v1.1 準拠）

**位置づけ:** RDD v1.1 の確定要件を、コンポーネント分解・技術選定・実装順序へ落とし込む。コード生成は本計画の承認後、コンポーネント単位の指示で着手する。

---

## 1. コンポーネント分解（7要素）

| # | コンポーネント | 責務 | 依存 |
|---|---|---|---|
| C1 | **検出コア**（純ロジック） | テキスト抽出 → チャンク分割 → 3層検出 → 和集合統合 → マスク。AWS外でも単体テスト可能な純関数群 | なし（boto3はC2で注入） |
| C2 | **検出アダプタ** | Guardrails(`ApplyGuardrail`)・Comprehend(`DetectEntities`) の呼び出しラッパ。リトライ・チャンク・スパン再マッピング | C1 |
| C3 | **処理関数ホスト** | C1+C2 をイベント起動で実行（Lambda）。S3入出力・ステータス更新 | C1,C2,C5 |
| C4 | **評価ハーネス** | 合成日本語データで recall/漏洩率/過剰マスク率を測定、ゲート判定（§7.4） | C1,C2 |
| C5 | **インフラ（IaC）** | S3・KMS(CMK)・VPCエンドポイント(S3/Bedrock/Comprehend)・IAM・EventBridge・Cognito・API Gateway | なし |
| C6 | **API** | 署名付きURL発行／ディレクトリ一覧／レビュー承認 | C5 |
| C7 | **Web UI** | アップロード・ディレクトリビュー・全件レビュー画面 | C6 |

> 設計意図：C1（検出コア）を AWS から切り離した純ロジックとすることで、recallの単体テスト・回帰テストをクラウド非依存で回せる。検出品質という本件の心臓部を、インフラやUIと独立に検証・改善できる構造にする。

---

## 2. 技術選定（推奨確定 ＋ 要判断の明示）

### 2.1 推奨で確定（簡潔性・AWSネイティブから一意）

| 領域 | 選定 | 根拠 |
|---|---|---|
| 処理言語 | **Python 3.x** | テキスト抽出（pypdf/pdfplumber, python-docx）・boto3・正規表現が揃い、Comprehend/Bedrock SDKと整合 |
| コンピュート | **Lambda（コンテナイメージ）** | 5ページ×3API呼出は軽量。scale-to-zeroで定常コスト≒0。15分上限内。抽出ライブラリが大きいためコンテナimage形式 |
| 検出呼出 | Guardrails=`ApplyGuardrail`（FM非呼出）、Comprehend=**チャンク同期`DetectEntities`** | 50文書/月では非同期ジョブは過剰。同期チャンクが簡潔 |
| 認証 | **Amazon Cognito** | マネージド、単一テナント・少数利用者に十分（D-7） |
| API | **API Gateway(HTTP API) ＋ Lambda** | 署名付きURL発行・一覧・承認の3エンドポイントに最小十分 |
| UIホスティング | **S3 ＋ CloudFront** | 静的配信。サーバ運用なし |
| リージョン | **東京（ap-northeast-1）** | データ所在を国内に。※Bedrock/Guardrails/Comprehendの当該リージョン提供可否はPhase 0で最終確認 |

### 2.2 要判断（3点のみ）

| ID | 論点 | 推奨 | 代替 |
|---|---|---|---|
| T-1 | IaC ツール | **AWS CDK（Python）** ― 処理言語と統一、VPCエンドポイント/KMS/Cognito等を型安全に記述 | SAM（サーバーレス特化で軽量）／Terraform（マルチクラウド志向） |
| T-2 | UIフレームワーク | **最小構成のReact SPA**（AWS SDK for JS／Amplifyライブラリでcognito・S3連携） | プレーンJS（さらに軽量だが保守性低） |
| T-3 | レビューUIの粒度 | **スパン単位の確認・追加・解除**（最小限） | 文書全体の一括承認のみ（簡潔だが見逃し補正力が弱い） |

> T-1〜T-3 はご指示の「簡潔さ優先で推奨どおり」に従い、無回答の場合は推奨で確定して進める。

## 3. 実装順序（後戻り最小化）

検出品質という最大の不確実性を先に潰し、秘匿性基盤を次に固め、UIを最後に積む順序とする。

| 段 | 作業 | 成果物 | ゲート |
|---|---|---|---|
| S1 | **Phase 0 第二段：実測ハーネス**（C4＋C1＋C2の最小形）。合成日本語データで3層のrecall/漏洩率を実測 | 実測レポート | recall実測が§7.4基準に到達見込みか判定。未達なら§6.4補強を先行検討 |
| S2 | **検出コア C1 ＋ アダプタ C2 の確立**。単体テスト整備 | テスト済み検出モジュール | 規則層recall=1.0、文脈依存≥0.98（合成データ） |
| S3 | **秘匿性基盤 C5**（S3＋CMK＋VPCエンドポイント＋IAM最小権限＋ログ抑止） | IaCスタック（基盤） | PrivateLink経路・暗号化・呼び出しログ無効の確認 |
| S4 | **処理関数ホスト C3**（Lambda化＋EventBridge起動＋ステータス管理） | 動作するバッチ処理 | アップロード→マスク→出力が一気通貫 |
| S5 | **API C6 ＋ 認証**（署名付きURL・一覧・承認、Cognito） | 認証付きAPI | 認証・署名付きURL短期失効の確認 |
| S6 | **Web UI C7**（アップロード・ディレクトリビュー・レビュー画面） | 動作するUI | 二経路アクセス・全件レビュー動作 |
| S7 | **評価ハーネス統合 C4**（本番構成でのゲート判定） | ゲート判定レポート | §7.4 Go条件の充足 |

> S1 を最初に置くのは、検出recallが基準に届かない場合、補強（§6.4）やタグ体系の見直しが S2 以降の前提を変えるため。最も手戻りの大きい検証を先頭に固定する。

## 4. コード生成着手前に必要なもの

1. **実測のためのAWS実行環境**：S1 のハーネスは実際の Guardrails／Comprehend 呼び出しを伴う。本環境からAWSへ到達できないため、ハーネスのコードは生成できても**実行は利用者のAWSアカウント**で行う。
2. **T-1〜T-3 の確定**（無回答なら推奨で確定）。
3. **コード生成の対象範囲の指示**：どのコンポーネント（推奨は S1 の実測ハーネス＝C4＋C1＋C2 最小形）から生成するか。

---

## 5. 直近の推奨アクション
S1（Phase 0 第二段の実測ハーネス）のコード生成から着手する。これは検出コア（C1）と検出アダプタ（C2）の最小実装を兼ねるため、S2 の土台にもなり、最も投資対効果が高い。生成後、利用者がAWSアカウントで実行して recall を実測し、その結果でゲート（§7.4）と補強要否を判断する。
