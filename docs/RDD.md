## 0. 本書の位置づけと v1.0 → v1.1 の変更

本書は、日本語文書に含まれる機密情報・個人情報を検出し、不可逆なタグ置換でマスキングするシステムの要件を定義する。利用者は**シンプルなWeb UI**から文書をアップロードし、結果を**S3 および Web UI のディレクトリビュー**から閲覧・取得する。実装コードは本書承認後に着手する。

**v1.1 の変更点（Phase 0 第一段の結果反映）：** Guardrails の組み込みPII型（全31種）には **ORGANIZATION / JOB_TITLE / FACILITY に相当する型が存在せず**、`NAME` 型は組織名を明示的に除外する。このため組織名・会社名・役職・施設名が現設計のままでは構造的に漏れることが判明した。これを埋めるため、**Amazon Comprehend の汎用エンティティ認識（日本語対応、ORGANIZATION / TITLE / LOCATION 等を返す）を和集合の第3検出層として追加**する。検出は **規則層 ∪ Guardrails ∪ Comprehend汎用NER の3層和集合** となる（旧 v1.0 は2層）。いずれもマネージドであり、**自前検出モデルは依然として構築しない**。

設計上の重心は二点に置く。

1. **捕捉性（recall）の構造的最大化** ― 単一の確率的検出器では漏洩ゼロを保証できない。3層和集合・全件人手レビュー・漏洩率を測定可能にする評価方法論で構造的に担保する。
2. **機密度に応じた秘匿性の写像** ― 情報の機密度を、処理経路・ログ・暗号化・保持期間という制御束へ対応づける。

## 1. 目的・スコープ

### 1.1 目的
日本語文書（PDF / Word / Markdown / プレーンテキスト）に含まれる個人情報・機密情報を、**マネージドサービスを主体とした最小構成**で検出し、**不可逆なタグ置換**でマスキングした成果物を生成する。利用者はWeb UIで完結し、結果をS3およびUIのディレクトリビューから確認できる。捕捉性（recall）の構造的最大化と機密度に応じた秘匿性を品質の中心に置く。

### 1.2 中核方針
- 検出は **規則層 ∪ Guardrails ∪ Comprehend汎用NER** の3層和集合。検出に用いるモデルはすべてマネージドであり、**自前検出モデルは構築しない**。
- 精度方針は **recall最優先**。FPは許容し、FN（情報漏洩）を構造的に最小化する。3層和集合と全件人手レビューで漏洩の実効値をゼロ近傍に抑える。
- 秘匿性は「クラウドからの完全隔離」ではなく、**情報の機密度に応じた制御の束**で担保する。全文書を最高階層相当で扱う（D-2）。
- 利用者接点は**シンプルなWeb UI**に限定し、UIは閲覧・操作層、S3を単一の真実のソースとする。

### 1.3 スコープ（In）
1. Web UI からの文書アップロード（PDF / Word / Markdown / プレーン）
2. テキスト抽出・正規化（OCR なし）
3. チャンク分割（API入力上限対応）＋スパン位置の文書全体マッピング
4. 規則層（正規表現・独立工程）による定型・日本固有識別子の捕捉
5. Guardrails 機微情報フィルタによる文脈依存PII（氏名・住所等）の捕捉
6. Comprehend 汎用NERによる組織名・役職・施設名等の捕捉
7. 3層スパンの和集合統合・重複解決
8. 不可逆タグ置換によるマスキング
9. 最高階層相当の秘匿性制御（全文書一律）
10. 全件人手レビュー工程
11. 結果への二経路アクセス：(a) S3、(b) Web UI ディレクトリビュー
12. 捕捉性・漏洩率の評価とゲート判定

### 1.4 スコープ外
- 自前検出モデルの学習・FT・常設推論エンドポイント（構築しない）
- OCR、リアルタイム同期API、可逆マスキング
- 元文書体裁を保持したマスク済み再生成（Phase B・保留 ― D-6）
- 下流処理（RAG・分析等。マスク済みS3を別途参照 ― D-1）
- 組織機能・ロール権限分掌（D-7）

## 3. 対象エンティティと機密度

### 3.1 規則層で捕捉（定型・日本固有識別子・独立工程）
メールアドレス、電話番号、**マイナンバー（12桁）**、**法人番号（13桁）**、銀行口座番号、クレジットカード番号、郵便番号、生年月日（和暦・西暦）、IPアドレス、URL、（必要に応じ）登記情報・各種公的番号。

> Phase 0 確定：マイナンバー・法人番号・日本形式銀行口座・郵便番号・生年月日は **Guardrails組み込み型・Comprehend汎用NERのいずれにも存在しない**。規則層がこれらの**唯一の捕捉手段**であり、その品質がrecallを直接決める。

### 3.2 Guardrails 機微情報フィルタで捕捉（文脈依存PII）
氏名（NAME）、住所（ADDRESS）、および組み込み型でカバーされるPII（EMAIL/PHONE/CARD/IP/URL 等。これらは規則層と二重捕捉）。日本語最適化済み・確率的。

### 3.3 Comprehend 汎用NERで捕捉（§3.2ギャップ補完・D-8）
**組織名・会社名（ORGANIZATION）、役職（TITLE）、施設名（LOCATION/ORGANIZATION）**、その他汎用NERが返す固有表現。日本語対応。Guardrailsが型を持たない領域を補完する第3層。

> 注：氏名・住所は Guardrails と Comprehend（PERSON/LOCATION）で二重捕捉されうる。和集合方針のもとで二重捕捉はrecall向上に寄与し、競合は§9「広い方優先」で解決。

### 3.4 機密度の扱い（全文書一律・最高階層相当 ― D-2）
検出前から平文を扱う以上、文書ごとの機密度判定は行わず、**全文書を最高階層相当の制御下に置く**。全件に対し、呼び出しログ無効化・中間生成物の即時削除・保持期間最短化・アクセス主体の限定を適用する。番号利用法上の特定個人情報（マイナンバー等）を含みうる前提を、システム全体の既定とする。

## 4. アーキテクチャ設計

### 4.1 基本思想
**Web UI（薄い閲覧・操作層）＋ S3（単一の真実のソース）＋ サーバーレス・バッチ処理 ＋ 3層和集合検出 ＋ 全件人手レビュー**。月50文書規模に対し、常設エンドポイント・自前モデル・重量オーケストレータ・常設ファイルシステムを排し、イベント起動の関数処理で完結させる。

### 4.2 構成

```
[Web UI (マネージド認証)]
   │  ① 署名付きURLでアップロード
   ▼
[S3 入力バケット (CMK暗号化, 最高階層制御)]
   │  ② S3イベント → EventBridge → 処理関数起動
   ▼
[処理関数 (Lambda／必要時 Fargate・VPC内)]
   1. テキスト抽出 : PDF/Word/MD/プレーン → 正規化テキスト（OCRなし）
   2. チャンク分割 : 各APIの入力上限（Guardrails／Comprehend 5000バイト）に合わせオーバーラップ分割、スパン位置を文書全体へマッピング
   3. 規則層       : 正規表現（独立工程）で定型・日本固有識別子を抽出
   4. Guardrails   : ApplyGuardrail（FM非呼出, 呼び出しログ無効）で文脈依存PII抽出
   5. Comprehend   : DetectEntities（日本語）で組織名・役職・施設名等を抽出
   6. スパン統合   : 規則 ∪ Guardrails ∪ Comprehend（和集合）、重複・包含解決（広い方優先）
   7. マスキング   : タグ置換（不可逆）
   │
   ▼
[S3 出力バケット (CMK暗号化)] ＋ [検出メタJSON]  … ステータス: レビュー待ち
   │  ③ UIで全件人手レビュー（確認・手動修正）→ 承認
   ▼
[確定済み成果物]  … S3 ＆ UIディレクトリビューから閲覧・取得
```

### 4.3 検出層の役割分担（3層）
- **規則層（独立工程）**：日本固有構造化PIIの唯一の捕捉手段。決定論的・即時・テスト容易（D-3）。
- **Guardrails**：氏名・住所等の文脈依存PII。FM非呼出の `ApplyGuardrail`、呼び出しログ無効。
- **Comprehend汎用NER**：組織名・役職・施設名等、Guardrailsが型を持たない領域（D-8）。
- **和集合統合**：3層スパンを結合、重複・包含は広い方を優先して残置漏れを防ぐ。

### 4.4 Web UI とアクセス経路（R-10〜R-12）
- **UI の役割**：アップロード・結果閲覧・人手レビューに限定した薄い層。処理ロジックを持たず、API 経由で S3 と対話。
- **認証**：マネージド認証（例 Amazon Cognito）で保護。単一テナント・少数利用者・権限分掌なし（D-7）。
- **アップロード**：API が **S3 署名付き URL** を発行し、ブラウザから入力バケットへ直接アップロード（ファイル実体を中継サーバに通さない）。
- **ディレクトリビュー**：S3 の論理プレフィックス階層（例 `yyyy/mm/文書ID/`）を**ディレクトリツリーとして表示**（`ListObjects` の階層化表示。常設ファイルシステムは持たない）。
- **人手レビュー画面**：マスク済みテキストと検出メタを表示し、未検出箇所の手動マスク追加・誤マスク解除を可能とする。承認で成果物確定。
- **二経路（R-11）**：(a) S3 直接、(b) UI ディレクトリビュー（署名付きURL経由）。同一S3実体への別ビュー。
- **状態表示**：アップロード済／処理中／レビュー待ち／確定済／失敗。
- UI ホスティングは静的配信（S3＋CDN）を基本とする。

### 4.5 ネットワーク・暗号化・監査
- 処理関数を VPC 内に配置。S3・Bedrock・**Comprehend** へは **PrivateLink（インターフェースVPCエンドポイント）** 経由とし、処理経路にインターネット egress を持たない。
- 入出力 S3・中間生成物を **CMK（KMS）** で暗号化。転送時 TLS。
- CloudTrail で操作監査。**§8 の呼び出しログ抑止に従い、処理ログに平文 PII を出さない**。
- 処理ジョブ実行ロールと UI/API ロールを分離し、最小権限のみ付与。

## 5. 秘匿性制御（全文書一律・最高階層相当）

D-2 に基づき全文書へ次を一律適用する。マネージド検出（Guardrails・Comprehend）も同一の制御下に置く。

- リージョン内処理 ＋ PrivateLink（S3・Bedrock・Comprehend）＋ CMK暗号化 ＋ CloudTrail監査。
- **Bedrock モデル呼び出しログの無効化**（原文残存の防止 ― §8）。
- **Comprehend の結果・入力をログ／永続化しない**運用（中間生成物として即時削除）。
- 中間生成物（抽出テキスト・チャンク・検出メタの作業コピー）の**処理完了後即時削除**。
- S3 ライフサイクルによる**保持期間の最短化**。
- アクセス主体の限定（認証済み少数利用者、最小権限 IAM）。

## 6. 検出戦略

### 6.1 規則層（独立工程）
日本固有識別子を中心に正規表現で決定論的に捕捉。チェックディジット検証は FP許容方針のもとで必須としない（recallを下げないため緩く広く捕捉）。

### 6.2 Guardrails 機微情報フィルタ
`ApplyGuardrail`（FM非呼出、呼び出しログ無効）。組み込み型のうち本件で有効なもの（NAME/ADDRESS/EMAIL/PHONE/CARD/IP/URL 等）を有効化。5000バイト上限に対しチャンク分割し、スパンを文書全体位置へ再マッピング。

### 6.3 Comprehend 汎用NER（第3層・D-8）
`DetectEntities`（日本語）で ORGANIZATION/TITLE/LOCATION 等を取得。リアルタイム上限5000バイト（日本語で約1600字）に対しチャンク分割。50文書/月規模では `DetectEntities` のチャンク逐次でも非同期 `StartEntitiesDetectionJob` でも可（実装時に選択）。

### 6.4 recall 補強（マネージド・規則の範囲に限定）
Phase 1 評価で漏洩率が基準未達のとき、簡潔な手段から順に補強する。**自前モデルは構築しない**。
1. 規則層のパターン網羅追加。
2. Guardrail のカスタム regex 追加。
3. Comprehend 検出スコア閾値の調整（recall側）。

## 7. 評価計画・合否基準（重心その一・D-4）

### 7.1 二段構えの合否ロジック
漏洩率の数学的ゼロは確率的検出器を含む系では保証できない。よって合否を二段で構成する。

- **第一段：機械検出のrecall基準（自動測定）**
  - 規則層エンティティ（定型・日本固有識別子）：**recall = 1.0 を要求**。
  - 文脈依存エンティティ（氏名・住所・組織名・役職・施設名・連絡先等）：**エンティティ別 recall ≥ 0.98 を目標基準**。
  - 漏洩率（真の機微スパンのうち機械未検出の割合）を測定し、改善対象を特定する。
- **第二段：全件人手レビュー（実効漏洩の吸収・D-5）**
  - 機械マスク結果を**全件レビュー**し、未検出箇所を手動マスク。承認で成果物確定。
  - これにより**確定成果物の実効漏洩率をゼロ近傍へ押し下げる**。月50文書・5ページで現実的に運用可能。

> 機械recall ≥ 0.98 は「人手レビュー負荷を実運用可能な範囲に保つための前提条件」であり、最終的な漏洩防止は全件レビューが担う。高い機械基準と簡潔な構成（自前学習なし）を、人手レビューの組み込みで両立させる。

### 7.2 副指標
precision（FP許容のため従指標）、過剰マスク率、処理成功率、レビュー工数（1文書あたり修正箇所数）。

### 7.3 評価データ
- 公開日本語データセットのホールドアウト（NER系公開コーパス等。ライセンスは Phase 0 検証）。
- 可能なら法務文書の代表サンプルに人手で機密スパンを付与したテストセット。**実PII不使用、合成または匿名化データで構成**。

### 7.4 ゲート（Go/No-Go）
- 比較：規則層のみ → ＋Guardrails → ＋Comprehend（3層）→ ＋§6.4補強。
- Go 条件：(i) 規則層エンティティ recall = 1.0、(ii) 文脈依存エンティティ recall ≥ 0.98、(iii) 全件レビュー工程が運用可能、(iv) 処理安定、(v) コスト規模相応。
- 未達時：§6.4 の補強を簡潔な手段から昇格。

## 8. セキュリティ非機能要件（重心その二）

| 区分 | 要件 |
|---|---|
| ネットワーク | 処理は VPC 内、S3・Bedrock・Comprehend へ PrivateLink、処理経路にインターネット egress なし |
| 暗号化 | 入出力 S3・中間生成物を CMK 暗号化。転送時 TLS。鍵アクセスを IAM 限定 |
| **呼び出しログ抑止** | **Bedrock モデル呼び出しログを無効化**（原文残存防止）。Comprehend 入出力も永続化しない。全文書一律 |
| 監査 | CloudTrail で操作監査。処理ログに平文 PII を出力しない（メタ情報のみ） |
| 最小権限 | 処理ジョブ実行ロールと UI/API ロールを分離。鍵・バケット・Guardrail/Comprehend 実行を分離管理 |
| UI/アップロード | 署名付き URL でブラウザ↔S3 直結。URL は短期失効。UI はマネージド認証で保護 |
| 保持期間 | 中間生成物を処理後即時削除。成果物も S3 ライフサイクルで失効 |
| 可用性 | バッチのため高可用要件なし。失敗時リトライで足りる |

## 9. マスキング仕様

- 方式：タグ置換（例 `田中太郎` → `[氏名]`、`株式会社○○` → `[組織名]`、`090-xxxx-xxxx` → `[電話番号]`）。
- 可逆性：**不可逆**。原文・マッピングを保持しない。
- 重複処理：3層和集合で同一・包含スパンが競合した場合、**広い方を優先**して残置漏れを防ぐ。
- 出力：マスク済み成果物（テキスト/MD/JSON）＋検出メタ JSON。レビュー承認で確定。
- Phase B（保留・D-6）：体裁保持の再生成は初期スコープ外。着手時は完了条件に「下層コンテンツの物理的除去（再抽出で復元不可）」を必須化。

## 10. リスクと対応

| # | リスク | 影響 | 対応 |
|---|---|---|---|
| R1 | Guardrails 組み込み型に日本固有識別子・組織/役職/施設型がない | recall低下・漏洩 | 日本固有識別子は規則層、組織/役職/施設はComprehendで補完（3層和集合） |
| R2 | Guardrails/Comprehend は確率的で閾値制御に限界 | 見逃し | 3層和集合＋全件人手レビューで構造的に補う |
| R3 | Bedrock/Comprehend のログに原文PIIが残存 | 漏洩・要件矛盾 | 呼び出しログ無効化・入出力非永続化を全件一律で必須化 |
| R4 | チャンク分割でスパン境界がずれる（2API分の上限） | 取りこぼし | オーバーラップ分割＋文書全体への再マッピングを両APIで統一 |
| R5 | テキスト抽出の読み順・崩れ | 文脈依存検出の精度低下 | 抽出品質を Phase 0 評価。崩れの大きい文書型は対象外と明示 |
| R6 | 公開データセットのライセンス不適合 | 利用不可 | Phase 0 で可否検証・代替確保。実PII不使用 |
| R7 | FN（見逃し＝漏洩） | 致命的 | 3層和集合・低閾値・全件人手レビュー |
| R8 | 人手レビュー工数の増大 | 運用負荷 | 機械 recall ≥ 0.98 で修正箇所を抑制。規模が小さく現実的 |
| R9 | 署名付きURL・UI経由の意図せぬ露出 | 漏洩 | URL短期失効、UIを認証保護、アクセス権最小化 |
| R10 | Comprehend汎用NERの過剰検出（一般名詞の誤組織判定） | 過剰マスク | FP許容方針で許容。用途(a)のため下流有用性問題は生じない |
